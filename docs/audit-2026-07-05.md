## 2. 审计方法

### 2.1 四阶段流程

```
阶段 0: App 特征侦察 → 语言/框架/架构画像 + 产品定位校准 + 历史修复盘点
```

- 代码定位：jcodemunch search_symbols → get_symbol_source (bounded) → Read (带 offset/limit)

- 禁止：逐文件全量 Read、无检查清单裸跑、style/nit 入报告

## 5. Phase 2.5 对抗反驳

### 5.1 方法

双 tier 反驳 agent（Opus + Sonnet），每条 CRITICAL/HIGH 经 5 维攻击：

1. **上游 guard** — 调用方是否存在 guard/assert？
2. **运行时可达性** — 路径在真实使用中是否可达？
3. **已有缓解** — 是否有现有代码缓解？
4. **严重性校准** — 产品定位下是否真的 CRITICAL/HIGH？
5. **触发概率** — 触发条件是否需要极不可能的操作序列？

**裁决规则**: 两 agent 一致 → 采纳；分歧 → 采纳更 skeptical 的（skeptic wins）。

### 6.2 MEDIUM 发现详情 (6 条)

#### M1 · A5-C1 · Admin 路由 Supabase 查询不解构 error · `app/api/admin/route.ts:22-33`

```typescript
// 现状: 6 处查询全部只解构 data/count
const { data } = await supabase.from("user_quotas").select("*, user_id")...
return NextResponse.json({ users: data ?? [] });  // data=null → 空数组, 不报错
const { count: users } = await supabase.from("user_quotas").select("*", { count: "exact", head: true });
// count 可能为 null, users ?? 0 = 0
```

- **复现**: RLS 配置错误 → Supabase 返回 `{data:null, error:{...}}` → `data??[]` 静默返回空数组 → 管理员面板显示全零
- **根因**: Supabase SDK 返回 `{data, error}` union 语义未被充分消费
- **建议**: 每处加 `if (error) { console.error(...); return NextResponse.json({error:...}, {status:500}); }`
- **defectType**: `semantic-misuse` · **triggerProbability**: `edge-case` · **existingMitigation**: `adminIds 白名单`

#### M2 · A5-H1 · Stripe Webhook 事件类型覆盖不足 · `app/api/stripe/webhook/route.ts:68-141`

```typescript
if (event.type === "checkout.session.completed") { ... }
if (event.type === "customer.subscription.deleted") { ... }
// 所有其他事件类型 → line 143-146 标记 processed=true 后丢弃
```

- **复现**: 用户订阅自动续费 → Stripe 发送 `invoice.payment_succeeded` → 被静默标记 processed → 无配额更新
- **根因**: 当前产品模型仅支持一次性 Checkout 支付，但未留扩展入口
- **建议**: 添加 `console.warn` 记录未处理事件类型；随产品扩展添加 `invoice.payment_failed`, `customer.subscription.updated` 等
- **defectType**: `incomplete-coverage` · **triggerProbability**: `low` · **existingMitigation**: `当前产品无其他触发路径`

#### M3 · A2-C2 · Spec 版本 INSERT+DELETE 无事务 · `lib/app-spec/spec-versions.ts:26-43`

```typescript
await supabase.from("spec_versions").insert({...});   // 成功
// ... 无事务
await supabase.from("spec_versions").delete()...       // 失败 → 旧版本堆积
```

- **复现**: INSERT spec 成功 → DELETE 因 RLS/网络超时失败 → 旧版本数超过 20 条上限 → 存储空间浪费
- **根因**: 非关键路径无防御意识
- **建议**: 用 PostgreSQL RPC 封装 INSERT + DELETE 为原子操作，或定期运行 `clean_old_versions()` 清理
- **defectType**: `atomicity-gap` · **triggerProbability**: `rare` · **existingMitigation**: `DELETE 失败不阻塞主流程`

#### M4 · A2-C1 · Spec 版本列名 version vs version_number 不匹配 · `lib/app-spec/spec-versions.ts:18` vs `supabase/migrations/20260625_q5_versioning.sql:19`

- **App 代码**: `SELECT version ... INSERT {version: nextVersion}` 读写 column `version`
- **Migration q5**: `UNIQUE(project_id, version_number)` + `trg_auto_version_number()` 操作 `version_number`
- **风险**: 如果生产 DB 列名是 `version`（来自 `sql/migrations/20260601_spec_versions.sql`），则 q5 的 UNIQUE 约束和触发器不生效 → A2-C1 原始发现重新成立
- **复现**: 在 Supabase SQL Editor 执行 `SELECT column_name FROM information_schema.columns WHERE table_name='spec_versions'` — 确认实际列名
- **根因**: 两次 migration 使用不同列名，`CREATE TABLE IF NOT EXISTS` 语义下后执行的被跳过
- **建议**: 统一列名 → 如果生产 DB 是 `version`，ALTER migration 约束到 `version`；如果是 `version_number`，改 app 代码
- **defectType**: `schema-mismatch` · **triggerProbability**: `medium` · **existingMitigation**: `无（需验证生产 DB）`

#### M5 · A3-C1 · execSync('flutter pub get') 无 timeout · `lib/sandbox/flutter.ts:31`

```typescript
execSync("flutter pub get", { cwd: projectDir });  // 无 timeout! 对比:
// 同文件 line 42-46:
execSync(cmd, { cwd: projectDir, timeout: timeoutMs });  // ✅ 有 timeout
```

- **复现**: Flutter pub 仓库不可达 → `flutter pub get` 挂起 → Inngest worker 被占用直到函数级超时
- **根因**: 拼写遗漏 — 同文件其他 execSync 调用正确使用了 timeout
- **建议**: `execSync("flutter pub get", { cwd: projectDir, timeout: 120000 })`
- **defectType**: `missing-timeout` · **triggerProbability**: `rare` · **existingMitigation**: `Inngest 函数级超时兜底`

#### M6 · A2-H3 · Stripe webhook maybeSingle null 静默跳过 · `app/api/stripe/webhook/route.ts:76-84`

```typescript
const { data: sub } = await supabase.from("subscriptions")
  .select("user_id").eq("stripe_customer_id", customerId).maybeSingle();
const userId = (sub as any)?.user_id as string | undefined;
if (userId) { /* upsert quota */ }
// userId undefined → 静默跳过, 无日志
```

- **复现**: 新用户首次 Checkout → subscriptions 表中尚无 stripe_customer_id → maybeSingle 返回 null → userId = undefined → 跳过配额升级 → 用户付了钱但没拿到 Pro
- **根因**: 用户注册与 Stripe Checkout 异步 — customer_id 绑定可能滞后
- **建议**: null 时记录 warn 日志 + 发送 Inngest 延迟事件重试绑定
- **defectType**: `observability-gap` · **triggerProbability**: `expected` · **existingMitigation**: `Stripe webhook 自动重试, 首次失败后最多重试 3 天`
