## 一、产品定位（一句话）

**AI 原生软件生产平台**：用户输入 App 想法，系统通过多智能体串行工作流生成完整项目方案（立项、PRD、计划、架构、UI、开发、测试、商业），结果持久化到 Supabase，支持导出与历史查看。

## 三、整体架构

```
用户浏览器
    ↓
Next.js App Router
    ↓
POST /api/projects/[id]/generate → prepareProjectWorkflow（置 running）→ inngest.send → 立即返回
    ↓
Inngest（本地 Dev Server / 云端）→ /api/inngest → generateProjectReport
    ↓
executeProjectWorkflow（8 Agent + LLM + agent_runs + final_report）
    ↓
lib/llm.ts → lib/supabase.ts → Supabase PostgreSQL
```

- **鉴权**：当前无用户登录；服务端仅用 `SUPABASE_SERVICE_ROLE_KEY`（**禁止**暴露到浏览器）。**公网部署前**须按 [安全审计与清单.md](./安全审计与清单.md) 补齐登录、`user_id`、RLS、限流等。
- **生成模式**：**异步（v1.2）**：`POST /generate` 仅入队；**本地须同时运行** `npm run dev` 与 `npx inngest-cli@latest dev`。

## 四、业务逻辑要点

1. **创建项目**：`idea` 非空、≥10 字、≤5000 字；自动生成短 `title`。
2. **启动生成**：
   - `pending`：直接开跑。
   - `failed`：确认后调用；工作流**删除**该项目旧 `agent_runs`，清空 `final_report`，再 `running`。
   - `completed`：默认 `POST` 不带 `forceRegenerate` 时工作流返回「已完成」不覆盖；**仅** `forceRegenerate: true` 时删 runs、清空报告、全量重跑。
   - `running`：**禁止**再次启动；返回 `WORKFLOW_ERROR_ALREADY_RUNNING`，HTTP **409**。
3. **单轮成功**：每条 Agent 插入 `agent_runs`（`running`）→ LLM → 更新 `completed` + `output`；全完后组装 `final_report`，`projects.status=completed`。
4. **失败**：当前 Agent 标记 `failed`，项目 `failed` + `error_message`。
5. **常量**：`lib/workflow.ts` 导出 `WORKFLOW_ERROR_ALREADY_RUNNING`，与 API 409 对齐。

## 七、环境变量（`.env.local`）

**大模型**：通过 **`OPENAI_BASE_URL` + `OPENAI_MODEL` + `OPENAI_API_KEY`** 接入任意 **OpenAI 兼容** 服务端；**首选国内模型**（DeepSeek、豆包、通义千问等），境外模型为可选项。变量名保留 `OPENAI_*` 仅为 SDK 兼容，值可为国内厂商 Key。

| 变量 | 用途 |
|------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase 项目 URL |
| `SUPABASE_SERVICE_ROLE_KEY` | 服务端读写（**保密**） |
| `OPENAI_API_KEY` | 大模型 API Key（兼容 DeepSeek / 豆包 / 千问等） |
| `OPENAI_BASE_URL` | OpenAI 兼容网关根地址（如 DeepSeek：`https://api.deepseek.com`） |
| `OPENAI_MODEL` | 如 `deepseek-chat`、厂商文档中的模型 id |
| `NEXT_PUBLIC_APP_URL` | 服务端 `fetch` 自有 API 用（端口与 dev 一致） |
