# App 生产工厂 — 开发纲要 & 开发目标

> **用途**：把「当前该做什么、不该做什么、怎么验收」浓缩在一处，供人机接力执行。  
> **详表与习惯**：见 [执行计划.md](./执行计划.md)；**技术路径与表结构**：见 [项目功能与架构交接.md](./项目功能与架构交接.md)；**多平台远景**：见 [多平台App生产工厂路线图.md](./多平台App生产工厂路线图.md)；**安全审计与清单**：见 [安全审计与清单.md](./安全审计与清单.md)。

## 一、产品定位（一句话）

**AI 原生软件生产平台**：用户输入 App 想法 → 多智能体串行产出完整**方案级**交付（立项、PRD、计划、架构、UI、开发、测试、商业）→ 落库、可导出与历史查看。  
远期路线是 **App Spec IR → 代码 → 构建检查 → 仓库**，见路线图；**不等于**当前 MVP 必达能力。

## 二、开发目标（分层）

### 2.0 模型与供应商策略（长期约束，必须做到）

**App 工厂调用大模型的优先级**：**国内/中文生态模型首选兼容**（包括但不限于 **DeepSeek、豆包（火山方舟）、通义千问** 等，以 **OpenAI 兼容 Chat Completions** 或项目内统一抽象接入）；**国外模型**（如 OpenAI、Anthropic 等）作为**同等可插拔能力**，排序在**之后**，便于合规、时延、成本与备用线路切换。  
**工程要求**：配置层支持 **`OPENAI_BASE_URL` + `OPENAI_MODEL` + 单一服务端 Key 变量**（当前为 `OPENAI_API_KEY` 命名兼容）；后续迭代中可扩展为「按 Agent / 按任务选模型」与多供应商路由，**不得**把工厂主线绑死在单一境外供应商上。

### 2.1 当前主线（MVP v1.2）

**目标**：在 **Inngest 异步队列** 下跑通端到端，且不破坏既有前端体验（轮询、导出、重试语义等）。

**可验收闭环**：

1. 前端创建项目 → 详情页触发 **`POST /api/projects/[id]/generate`**（秒回 `mode: "async"`）。
2. **`inngest.send`** 投递 **`project/generate.requested`**，Inngest Dev / 云能消费。
3. 后台 **`executeProjectWorkflow`** 串行跑 **8 个 Agent**。
4. **Supabase** 更新：`projects` 至 `completed`，`final_report` 有内容；**`agent_runs` 共 8 条**且成功路径下均为 `completed`（表名以 `sql/schema.sql` 为准）。
5. 前端轮询展示结果，无「长 HTTP 卡死」；`running` 重复请求 **409** 等行为符合设计。

### 2.2 中期（v1.2 合并 main 之后）

按 [执行计划.md](./执行计划.md) 阶段表推进：可观测与成本（v1.3）→ 代码生成与 ZIP（**v2a** 可安装 App、**v2b** 微信原生小程序必选）→ 沙箱与自动修（v2.1）等；**每一项单独验收**，不提前捆绑。

### 2.3 长期（多平台「生产工厂」）

**App Spec IR**、Flutter / 鸿蒙 / 后端三栈、**微信原生小程序（必选交付，与可安装 App 并列，非可选项）**、模板矩阵、构建与自动修复——见 [多平台App生产工厂路线图.md](./多平台App生产工厂路线图.md)。  
**阶段判断**：路线与 Spec **文档**已就位 ≠ **机器可执行** Schema / 生成器 / 表结构已就绪；实现需分阶段立项；**小程序能力未上线时须对外诚实标注「未就绪」，不得写成「不需要小程序」**。

## 三、当前阶段优先级（硬顺序）

1. **唯一最高优先级**：**v1.2 双进程本地联调**（Next + Inngest Dev，端口与 `/api/inngest` 一致）。
2. 联调 **通过** 后，再议：合并分支、上云 Inngest 密钥、下一阶段功能。
3. 联调 **通过** 后，再启动：**App Spec v0.1 的 JSON Schema**（可放在 `docs/schemas/` 或后续 `packages/`），**仍不接**主业务流程，直到单独评审。

---

## 四、安全架构与落地路线（前期 · 中期 · 后期）

**原则**：安全不取决于「网页 vs App」，而取决于 **HTTPS、登录与鉴权、密钥仅服务端、DB 行级隔离（RLS）、生成与后台任务防滥用、日志不泄密、备份**。网址是门牌号；**账号、权限、Service Role、模型 Key** 才是门锁与保险柜。算力：**编排在你部署的 Next 上，推理在模型服务商 API，数据在 Supabase**。

| 阶段 | 目标 | 必须做到的事（摘要） |
|------|------|----------------------|
| **前期（规划 + 当前开发）** | 底线不踩穿；风险可陈述 | 密钥不进前端、不 `NEXT_PUBLIC_` 泄露 AI/Service Role、`.env` 不进 Git、AI **只走后端**、本地/内网调试习惯；承认 **当前 MVP 无登录 = 不适合公网裸奔**。详见 [安全审计与清单.md](./安全审计与清单.md) §三、§4.1。 |
| **中期（v1.2 联调通过后）** | 可审计、可设计、可排期 | 只读安全审计报告；设计 **Supabase Auth**；`projects` 等表 **`user_id`**；**RLS** 策略与迁移落盘；所有项目相关 API **校验资源归属**；**Inngest** 生产 Signing + 业务鉴权对齐；**限流 / 额度**。§4.2。 |
| **后期（上线前必须落地）** | 生产级可信 | 正式 **HTTPS**；RLS 实测通过；浏览器侧 **仅用 anon + JWT**（禁止浏览器持 Service Role）；生成接口 **登录 + 额度 + 频率**；Inngest 异常监控；**DB 备份**与恢复演练；若有管理端则 **role + 审计 + MFA**；**隐私政策**。§4.3。 |

**与路线文档的关系**：本节是**执行级安全纲要**；逐项勾选与当前代码审计结论以 [安全审计与清单.md](./安全审计与清单.md) 为准，随 Auth/RLS 上线后更新该文档 §三。

## 五、约束（承接执行，避免跑偏）

在 **v1.2 联调未稳定通过** 前，**默认不做**：

- 不实现 **App Spec** 的 JSON Schema / TS 类型 / Validator / 样例 Spec / 错误码规范（除非单独开任务且不影响联调）。
- 不上 **Flutter / Harmony** 生成器、不大改 **数据库** 为多平台工厂表、不接 **构建沙箱**、不上 **自动修复 Agent**。
- 不把 **v1.2 主流程** 改成「必须先过 App Spec」的强依赖。
- 不继续 **扩展** 多平台说明文档篇幅（小修链接、索引即可）。

**允许**：仅文档路径/索引修正；联调相关的**必要小修复**（先复现、再改、可回滚）；**安全相关**允许：文档、示例环境文件警示、服务端模块注释、不改变对外行为的纯安全加固（须单独评审）。

## 六、v1.2 联调要点（精华）

| 类别 | 要点 |
|------|------|
| **环境** | `NEXT_PUBLIC_SUPABASE_URL`、`SUPABASE_SERVICE_ROLE_KEY`、`OPENAI_API_KEY` 必填；`NEXT_PUBLIC_APP_URL` 建议与真实 Origin 一致；本地 **`INNGEST_*` 可留空** 配合 CLI。 |
| **进程** | 终端 A：`npm run dev:3000`（或同端口 `dev`）；终端 B：`npm run inngest:dev`（默认指向 `http://localhost:3000/api/inngest`，端口变了要改 `-u`）。 |
| **路由** | Inngest：`app/api/inngest/route.ts`；触发：`app/api/projects/[id]/generate/route.ts` → `inngest.send`。 |
| **排错** | `running` 无 runs：查 Inngest 控制台与 Dev URL；投递失败看 `projects.error_message`；非 `running` 的过期事件应 **skipped** 而非误标 **failed**（见当前 `workflow` 设计）。 |

更细的命令与验收 SQL 见 [README.md](../README.md) 与 [版本与分支.md](./版本与分支.md)。
