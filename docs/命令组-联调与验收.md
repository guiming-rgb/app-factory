# 命令组 — 联调与验收（章程配套）

> **原则**：能自动做的命令 **按组一次性交给 Agent 执行**（单条依次或明确可并行的并列），避免维护者「一条一条自己敲」。  
> **上位规则**：[开发纲要.md](./开发纲要.md) §二点五、§二点六 · [维护者最小参与清单.md](./维护者最小参与清单.md)

---

## 一、编组规则（Agent 必守）

| 规则 | 说明 |
|------|------|
| **组大小** | 每组 **4～6 条** 命令（含 `cd` 进目录）；不足 4 条不单独成组，合并到相邻组 |
| **一次交付** | 对用户说「执行 **G2**」时，Agent **在同一轮尽量跑完整组**，再汇报 ✅/❌，不要拆成 6 次对话各跑一条 |
| **顺序默认** | 组内 **从上到下依次**执行；标「可并行」的可在同一轮 **多条独立 Shell 并行** |
| **后台长驻** | `start` / `inngest:dev:3001` 用 **后台 Shell**；组末用 `curl` / `npm run accept` 验证 |
| **维护者** | **不**按组自己敲；仅当 Agent 写明「需您 G0-② 填 .env」等万不得已项 |

**组 ID**：`G0`～`G5`（下表）；新阶段可增 `G6+` 并在本文件追加。

## 二、标准命令组（复制给 Agent 或维护者旁观）

### G0 — 环境自检（4 条 · 顺序 · 秒级）

维护者新机/新会话时 Agent 先跑；**不启服务**。

```bash
cd "/Users/guiming/Desktop/app生产工厂/app-factory"
test -f .env.local && echo "✓ .env.local" || echo "✗ 缺 .env.local"
grep -E '^(INNGEST_DEV|NEXT_PUBLIC_APP_URL)=' .env.local 2>/dev/null | sed 's/=.*/=…/' || true
git status -sb | head -5
node -v && npm -v
```

**失败时**：维护者仅补 **`.env.local`** 或 Git，勿继续 G3/G4。

---

### G1 — 读库验收（4 条 · 顺序 · 需 `.env`）

**不**要求 3001/Inngest 已启动；查 Supabase 与最近项目。

```bash
cd "/Users/guiming/Desktop/app生产工厂/app-factory"
npm run verify:v13
npm run accept
git log -1 --oneline
ls -la sql/migrations/20260519_usage_logs.sql 2>/dev/null || true
```

**通过标志**：`accept` 末尾 **✅ accept 通过** 或 `verify:v13` **8/8**。

---

### G2 — 工程门禁（5 条 · 顺序）

改代码后、合并前、收工前。

```bash
cd "/Users/guiming/Desktop/app生产工厂/app-factory"
npm run lint
npm run build
npm run accept
curl -s -o /dev/null -w "home HTTP %{http_code}\n" http://localhost:3001/ || echo "（3001 未启，可忽略）"
git status -sb
```

**通过标志**：`lint`/`build` 无 ERROR；`accept` ✅（若 3001 未启，home 行可忽略，由 G3 负责起服）。

---

### G3 — 双进程启动（5 条 · 顺序 + 2 后台）

**Agent 操作**：①～② 前台；③④ **后台 Shell**；⑤ 健康检查。

```bash
cd "/Users/guiming/Desktop/app生产工厂/app-factory"
lsof -ti :3001 | xargs kill 2>/dev/null || true
npm run build
# ↓ 以下两条由 Agent 各开「后台终端」执行，不要关：
#   npm run start -- -p 3001
#   npm run inngest:dev:3001
sleep 5
curl -s -o /dev/null -w "site %{http_code}\n" http://localhost:3001/
curl -s -o /dev/null -w "inngest-route %{http_code}\n" http://localhost:3001/api/inngest
```

**通过标志**：site **200**；inngest-route **200** 或 **405**（GET 探测常见）。

---

### G4 — 完整联调验收（6 条 · 顺序 · 含生成）

在 **G3 已起双进程** 后执行；`PROJECT_ID` 换成实际 UUID。

```bash
cd "/Users/guiming/Desktop/app生产工厂/app-factory"
PROJECT_ID="833ad678-f204-40d7-a47c-5b76e803f64f"
curl -s -X POST "http://localhost:3001/api/projects/${PROJECT_ID}/generate" \
  -H "Content-Type: application/json" \
  -d '{"forceRegenerate":true}'
npm run verify:v13 -- "${PROJECT_ID}"
sleep 90
npm run verify:v13 -- "${PROJECT_ID}"
npm run accept -- "${PROJECT_ID}"
curl -s "http://localhost:3001/api/projects/${PROJECT_ID}/usage" | head -c 400
echo ""
```

**说明**：两次 `verify:v13` 中间 `sleep 90` 可改为 Agent 轮询直至 8/8 或超时 20min。  
**通过标志**：最终 `accept` ✅ 且 `usage` JSON 中 `usage_logs_row_count` ≥ 8。

npm run build 2>&1 | tail -3
git status -sb

## 三、常见场景 → 用哪几组

| 场景 | 命令组（Agent 顺序跑） |
|------|------------------------|
| 新会话、不确定环境 | **G0** → **G1** |
| 只改了文档 | **G1** 或 **G5** |
| 改了 TS/业务代码 | **G2**；要真跑流程再 **G3** → **G4** |
| 维护者说「验收 v1.3」 | **G3** → **G4**（无 PROJECT_ID 则 G4 用 G1 自动最近项目） |
| 收工 | **G2** 或 **G5** + 更新文档 |

**不要**把 G0～G5 合成一条超长 shell 链（除 G4 内已写的顺序外）；长驻进程必须后台。

## 四、维护者只需知道

- 对你说：**「跑 G2」** 或 **「G3+G4 验收踢足球项目」** — Agent 按组执行并贴 ✅/❌。  
- **不必**自己把组里 5～6 条命令逐条敲进终端。
