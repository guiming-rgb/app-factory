### 2026-05-14 正式收工备忘（流程 · 本地 URL · 项目建立 · 验收 / 数据检疫）

#### 1. 环境准备（一次性 / 换机时）

1. 目录：`app-factory/` → `npm install`。  
2. Supabase：在控制台建库后，执行仓库内 **`sql/schema.sql`**、**`sql/seed.sql`**（表结构真源见 [开发纲要.md](./开发纲要.md) 与 [项目功能与架构交接.md](./项目功能与架构交接.md)）。  
3. `cp .env.local.example .env.local`，按示例填入 **`NEXT_PUBLIC_SUPABASE_URL`**、**`SUPABASE_SERVICE_ROLE_KEY`**、**`OPENAI_*`**；**`NEXT_PUBLIC_APP_URL`** 与 Next 端口一致（默认 `http://localhost:3000`）。  
4. **本地 Inngest 必加**：`.env.local` 中 **`INNGEST_DEV=1`**，否则易出现 cloud 模式无密钥、**`PUT /api/inngest` 500**（说明见 [README.md](../README.md)）。

#### 2. 双进程运行与涉及网址（本地）

| 用途 | 地址或命令 |
|------|------------|
| **工厂 Web（用户界面）** | **`http://localhost:3000`**（首页创建想法；`/projects` 历史；`/projects/[id]` 详情） |
| **Inngest 同步端点（给 Dev Server 连）** | **`http://localhost:3000/api/inngest`**（若 Next 改用 3001，则改为 `http://localhost:3001/api/inngest`，与 [版本与分支.md](./版本与分支.md) 一致） |
| **终端 A（Next）** | 建议 **`npm run dev:3000`**（`package.json` 固定 3000 端口） |
| **终端 B（Inngest Dev）** | **`npm run inngest:dev`**（等价于 `npx inngest-cli@latest dev -u http://localhost:3000/api/inngest`） |
| **Inngest 本地控制台（常见）** | CLI 启动后终端会打印，多为 **`http://localhost:8288`** — 用于确认函数已注册、事件被消费 |

#### 3. 从「建项目」到「生成完成」的产品流程（与代码路径对齐）

1. 浏览器打开 **`http://localhost:3000`**，在首页提交 **App 想法**（校验：**≥10 字、≤5000 字**，见 [项目功能与架构交接.md](./项目功能与架构交接.md) §四）。  
2. 前端 **`POST /api/projects`** → 写入 **`projects`**，`status=pending`，跳转 **`/projects/[id]`**。  
3. 在详情页点击 **「启动 AI 后台生产」** → **`POST /api/projects/[id]/generate`**：  
   - **v1.2**：`prepareProjectWorkflow` 将项目置 **`running`**，**`inngest.send`** 投递事件 **`project/generate.requested`**，接口**立即返回** `{ success: true, mode: "async" }`（见 [README.md](../README.md)）。  
4. **须保持终端 B（Inngest Dev）运行**；Inngest 调用本应用的 **`/api/inngest`** → 执行 **`executeProjectWorkflow`**，按固定顺序跑 **8 个 Agent**（`ceo` → … → `business_advisor`，见交接文档 §六）。  
5. 详情页 **`running`** 时自动轮询刷新；完成后 **`status=completed`**，可查看 **`final_report`**、导出 Markdown、复制报告等（v1.1 能力保留）。  
6. **并发语义**：`running` 再次 `POST generate` → **409**；已完成且未带 **`forceRegenerate: true`** → **400**（见 README API 说明）。

#### 4. 验收 A + 数据检疫（Supabase 侧核对，建议收工 / 恢复时各跑一次）

以下为**逻辑验收**（在 **Supabase → SQL Editor** 执行；将下面三处 **`'<你的 projects.id>'`** 换成详情页 URL 里的项目 UUID）。

```sql
-- A. 项目态：应为 completed，且 final_report 非空
select id, status, length(coalesce(final_report, '')) as final_report_len
from projects
where id = '<你的 projects.id>';

-- B. Agent 条数：成功路径应为 8 条（可按需加 and status = 'completed'）
select count(*) as agent_run_rows
from agent_runs
where project_id = '<你的 projects.id>';

-- C. 各 Agent 状态分布（排错用）
select agent_id, status, count(*) as n
from agent_runs
where project_id = '<你的 projects.id>'
group by agent_id, status
order by agent_id;
```

**产品侧验收**（与 [执行计划.md](./执行计划.md) §五「验收 A」一致）：`projects.status=completed`；**`final_report` 有实质长度**；**8 条 `agent_runs`** 在成功路径下均为 **`completed`**；无异常重复 runs（除主动 **`forceRegenerate`**）；导出 / 复制 / 历史列表 / **409** 行为符合 [项目功能与架构交接.md](./项目功能与架构交接.md)。

## 三、完整执行计划（从现状到「App 生产工厂」长期目标）

### 3.1 愿景（终点形态）

用户从 **App 想法** 出发，在平台内完成：**方案 → 代码 → 测试修复 → 仓库/部署 →（可选）运营迭代**，多智能体 + 工具 + 沙箱 + 评测 + 人工审核节点，**可替换模型与执行器**，不绑死单一框架。

### 3.2 原则

- **模板 + 评测 + 人类在环**：关键节点可确认；质量可观测。
- **可插拔**：编排 / 模型 / 工具（含 MCP）可演进。
- **先闭环再扩张**：每一阶段有可演示的端到端结果。

**表注（虚拟汇总行）**：**MVP v2** 无独立排期与 PR 粒度；**状态**列 **—（由 v2a、v2b 推导）** 表示不单独维护「进行中」标记。若希望表内一眼可见，可在 **v2a、v2b 均 ✅** 后将本行状态改为 **✅**。

**关于「小程序必选」的难度与是否拖慢 App（拍板级说明）**：

- **难度**：首版不小——小程序是 **独立技术栈**（WXML/WXSS/JS 或跨端编译方案）、**微信登录与类目审核**、**域名与 HTTPS 配置**、与 App 不同的 **发布与合规**；工厂侧还要维护 **单独模板与能力矩阵**（见 [多平台App生产工厂路线图.md](./多平台App生产工厂路线图.md)）。  
- **是否影响正常 App 开发**：**设计上不应互相阻塞**——与路线图一致，采用 **统一 App Spec + 分栈生成器**（Flutter Gen / 小程序 Gen **并行分支**），共享 **后端契约**（如 Supabase / API）；**先完成 v2a、再完成 v2b**（或并行）均可，但 **v2b** 在表里为 **独立必达行**（未就绪时对外标「能力未上线」，而非「不需要」）。  
- **结论**：工作量会增加，但通过 **Spec 先行、生成器分阶段、队列步骤命名空间隔离**，避免「为了小程序改坏 App 模板」的耦合风险。

## 五、分阶段说明（要做什么 / 验收什么）

### 阶段：验收 A（真机联调）

- **做**：配置 `.env.local`；执行 `schema.sql` / `seed.sql`；`npm run dev` 跑完整生成；测 v1.1 功能；可选错误 Key 测 failed 重试。
- **验收**：`projects.status=completed`、`final_report` 有长度、8 条 `agent_runs` 均为 `completed`、无重复 runs（除主动 forceRegenerate）、导出/复制/历史/409 符合预期。
- **产出**：验收记录（可写在「二、上次进展」或单独 [`docs/验收记录.md`](./验收记录.md)）。

### 阶段：MVP v1.2（异步队列 — 建议 Inngest）

- **做**：引入 Inngest（或 Trigger.dev）；`POST /generate` 只投递事件并返回 `jobId`/`accepted`；函数内 `step.run` 按 Agent 拆分；失败策略与幂等；DB 状态与现有一致。
- **验收**：关闭浏览器后任务仍完成；无长时间阻塞 HTTP；失败可重试且不破坏数据一致性。
- **依赖**：验收 A 通过后再做，避免异步层掩盖配置问题。

### 阶段：MVP v1.3（可观测与成本）

- **做**：`usage_logs` 或等价表、每步耗时、可选 token 统计；简单查询或页面。
- **验收**：能回答「单次生成花多久、调了几次模型」。

### 阶段：MVP v2a（代码生成 — 可安装 App）

- **做**：固定技术栈模板；开发 Agent 输出文件树 + 写入存储或对象存储；ZIP 下载；可选 GitHub OAuth + push；可安装客户端（如 Flutter）最小模板 + 生成 + 本地 `analyze`/`build` 在约定环境下通过。  
- **验收**：下载的 App 工程可本地 `install && build`（在约定模板下）。  
- **与 v2b / 总阶段关系**：v2a **可先于** v2b 验收；**「代码生成」大阶段对外收官**以里程碑表 **MVP v2（总阶段·虚拟汇总）** 行为准：**v2a + v2b** 均达最小验收。

### 阶段：MVP v2b（代码生成 — **微信原生小程序**，必选）

- **做**：同一套 **App Spec** 输入；微信原生小程序独立模板 + Generator；产物 ZIP/目录可导入微信开发者工具；构建命令与登录/后端对接范围以 [多平台App生产工厂路线图.md](./多平台App生产工厂路线图.md) 模板矩阵为准。  
- **验收**：生成的小程序工程在约定微信开发者工具版本下 **可编译通过**（首版不要求全功能 parity，但不得仅为空壳占位且无构建路径）。  
- **与总阶段关系**：**MVP v2** 虚拟汇总行的完成条件含本节验收通过。  
- **依赖**：建议 **App Spec v0.1** 已能表达小程序所需字段（或与 v2a 共用同一 Spec 版本）；若 Spec 未含小程序段，须在里程碑备注中写明 **临时契约** 与补齐计划。

### 阶段：MVP v2.1～v3

- **做**：Docker/隔离沙箱、测试命令、错误回传迭代；一键预览部署（如 Vercel）。
- **验收**：从想法到可访问预览链接（在约定范围内）。

### 阶段：v4～v6

- **做**：登录与权限、记忆与技能版本化、工具/MCP 与执行器插件化。
- **验收**：按产品定义分项验收（上线前须合规与隐私评审）。
