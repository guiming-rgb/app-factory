## CRITICAL（5 条 — 上架前必修）

### C1 · OpenAI/DeepSeek 客户端无超时配置
- **证据**: `lib/llm.ts:55-62` — `new OpenAI({ apiKey, baseURL })` 缺 `timeout`，默认 600s
- **复现**: DeepSeek API 接受 TCP 连接但不返回响应 → 调用阻塞最多 10 分钟
- **根因**: OpenAI SDK 构造函数未传 `timeout` 或 `signal`
- **修复**: 添加 `timeout: 120000`（2 分钟）+ `maxRetries: 1`
- **缺陷类型**: resource-leak | **触发概率**: low | **缓解**: circuit breaker 切换备用端点但不断开挂起连接

### C2 · LLM 调用无 AbortController / 请求取消
- **证据**: `lib/llm.ts:214-227` — `callLLM()` 不创建也不接受 `AbortSignal`
- **复现**: 用户关闭浏览器 → serverless 函数继续阻塞在 LLM 调用直到 30s 超时
- **根因**: 未集成 AbortController
- **修复**: `callLLM` 接受 `signal?: AbortSignal`，用 `Promise.race` 包裹
- **缺陷类型**: resource-leak | **触发概率**: medium | **缓解**: 无

### C3 · verify-artifact.ts 死代码 — 生成 ZIP 零验证
- **证据**: `lib/codegen/verify-artifact.ts::verifyGeneratedArtifact` 定义了但从零个生产路径调用
- **复现**: 生成任意 codegen 输出 → ZIP 不经结构验证直接返回用户
- **根因**: 验证函数写完但从未接入 pipeline（BaseCodegenExecutor、Inngest handler、API route）
- **修复**: 在 `BaseCodegenExecutor.execute()` 的 finally 块中调用 `verifyGeneratedArtifact`
- **缺陷类型**: missing-validation | **触发概率**: always | **缓解**: 无

### C4 · codegen_runs 状态转换无守卫
- **证据**: `lib/codegen/runs.ts:89-93` — `updateCodegenRun` 接受任意 status，不检查当前状态
- **复现**: `markCodegenRunRunning(completedRunId)` → "completed" 被覆盖为 "running"
- **根因**: 状态机无守卫转换。UPDATE 应用 `.eq("status", expectedStatus)` 条件
- **修复**: `updateCodegenRun` 加 `currentStatus` 参数，UPDATE 加 `.eq("status", currentStatus)`
- **缺陷类型**: conditional-race | **触发概率**: edge-case | **缓解**: 无

### C5 · 多 execSync/spawnSync 无 timeout（超越 M5）
- **证据**: 
  - `lib/sandbox/flutter.ts:33` — `execSync("dart analyze")` 无 timeout
  - `lib/sandbox/flutter.ts:64` — `execSync("docker build")` 无 timeout（首次 10-20 分钟）
  - `lib/sandbox/flutter.ts:79` — `execSync("docker run")` 无 timeout
  - `lib/sandbox/flutter.ts:25,56` — `spawnSync("docker", ...)` 无 timeout
- **复现**: Docker daemon 无响应 → 子进程永久挂起
- **根因**: 部分 execSync/spawnSync 遗漏 `timeout`（与有 timeout 的 `runFlutterApkDebugBuild` 不一致）
- **修复**: 所有 execSync/spawnSync 统一加 `timeout: 120000`
- **缺陷类型**: resource-leak | **触发概率**: medium | **缓解**: 部分已有 timeout（不一致）

## HIGH 精选（10 条代表性发现）

### H1 · prepareProjectWorkflow 项目状态 TOCTOU
- **证据**: `lib/workflow.ts:66-96` — 读状态 + 更新状态无原子性，两个并发请求可同时进入执行
- **修复**: UPDATE 加 `.neq("status", "running")` 条件

### H2 · createCodegenRun 查重-插入竞态
- **证据**: `lib/codegen/runs.ts:30-52` — SELECT 检查 + INSERT 两步非原子
- **修复**: 加 `UNIQUE(project_id, target) WHERE status IN ('queued','running')` 约束

### H3 · Stripe Webhook 多表写入无事务
- **证据**: `app/api/stripe/webhook/route.ts:87-102` — subscriptions.upsert + user_quotas.upsert 分离
- **修复**: 用 Supabase RPC 函数包裹为单次原子操作

### H4 · Inngest Functions 无并发限制
- **证据**: `lib/inngest/functions.ts:12-35` — 4 个 function 缺 `concurrency` 配置
- **修复**: 加 `concurrency: 3` 防止 LLM API 配额耗尽

### H5 · CSP 使用 unsafe-inline + unsafe-eval
- **证据**: `middleware.ts:17` — `script-src 'self' 'unsafe-inline' 'unsafe-eval'`
- **修复**: 替换为 nonce-based CSP

### H6 · API 限流为全局计数器非 per-IP
- **证据**: `middleware.ts:24-31` — 单一全局计数器，一个用户 60 次请求触发全员 429
- **修复**: 分桶键改为 IP 或 user ID

### H7 · getUserQuota 无错误检查致 Pro 降级
- **证据**: `lib/auth/quota.ts:30,36,47-49` — `.select()/.insert()` 不解构 error
- **修复**: 检查 error 并 propagate，不静默 fallback 到 free tier

### H8 · WeChat/Harmony executor 无错误监控
- **证据**: `lib/codegen/execute-wechat.ts:144-148` + `lib/codegen/execute-harmony.ts:144-148` — catch 块只调 `markCodegenRunFailed` 不调 `captureError/notifyChannel`
- **修复**: 对齐 `executeFlutterCodegen` 的监控模式

### H9 · spec_versions 多查询无事务
- **证据**: `lib/app-spec/spec-versions.ts:28,43` — INSERT + DELETE 分离无事务
- **修复**: 用 Supabase RPC 包裹

### H10 · Stripe Webhook 幂等检查有竞态窗口
- **证据**: `app/api/stripe/webhook/route.ts:44-65` — upsert 后 re-read 之间无锁
- **修复**: 使用 `SELECT FOR UPDATE` 或 advisory lock

## 方法反思

### 本次 v3 方法证明有效
1. **穷举矩阵**（Phase 0.8）在 Agent 启动前就发现了 API 鉴权盲区和模板断链
2. **覆盖追踪**（Phase 1.6）暴露了 `.github/` 和 `templates/` 的 0% 覆盖率
3. **交叉反驳**（Phase 2.5）剔除了 3 条虚构发现——包括 2 条 CRITICAL——节省了修复错误代码的时间

### Agent 虚构问题（DeepSeek 专属风险）
A6（安全）的 2 个 CRITICAL 都是虚构的：
- "Stripe webhook 无签名验证" — 代码实际有 `constructEvent()` 调用
- "API key 明文存储" — 代码实际有 SHA-256 哈希管道

**教训**: 安全维度的 Agent 容易"看到安全风险"的倾向过强，即使代码已正确处理。反驳 agent 必须是不同模型，且必须 Read 回读确认。

### 补充覆盖建议
- `.github/workflows/` 应独立审计（本次仅 17% 覆盖）
- `lib/wechat-codegen/emit-*.ts` 和 `lib/harmony-codegen/emit-*.ts` 的 emit 函数应专项抽查
- Supabase migrations 应全量对照 TypeScript 代码做列名校验
