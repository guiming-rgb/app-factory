# 维护者最小参与清单

> **原则**：调试验收 **默认不由您点网页**；费时间、易出错（错端口、错 Supabase 项目、漏 build、手敲项目 ID）。  
> **Agent 默认执行**：按 **[命令组-联调与验收.md](./命令组-联调与验收.md)** 的 **G0～G5** 成组跑（每组 4～6 条命令），含 `accept`、build、双进程等。  
> **您可说**：「跑 G2」「G3+G4 验收」— 不必自己逐条敲命令。  
> **您只做下面「万不得已」项。**

---

## 您通常不用做

- ❌ 反复开/关两个终端自己猜命令  
- ❌ 在详情页找「v1.3」或版本号  
- ❌ 手敲 `localhost:3001/projects/...` 长 ID  
- ❌ 在 Supabase 里对账（Agent 用 `verify:v13` / `accept` 读库）  
- ❌ 打开 `https://xxx.supabase.co` 当网页  

---

## 您只需要做（最多 3 件）

| # | 何时 | 做什么 | 频率 |
|---|------|--------|------|
| 1 | 新项目/新机器 | 复制 `.env.local.example` → `.env.local`，填 Supabase URL、Service Role、OpenAI Key；`INNGEST_DEV=1`，`NEXT_PUBLIC_APP_URL=http://localhost:3001` | 一次 |
| 2 | 首次上 v1.3 或新 Supabase 项目 | 在 **与 .env URL 一致** 的 Supabase → SQL Editor → 跑 `sql/migrations/20260519_usage_logs.sql`（无 `projects` 则先 `schema.sql`） | 一次/每库 |
| 3 | 可选 | 愿意看界面时：打开 `http://localhost:3001` 看一眼流程是否顺眼 | 按需 |

---

## 您如何知道「过了没有」

**不用自己判断。** 看 Agent 发来的终端最后一行即可：

```text
✅ accept 通过（终端验收完成，维护者无需再点详情页）
```

或：

```text
✅ v1.3 通过：数据库里已有用量记录。
```

若是 ❌，**把整段终端输出转发给 Agent**，由 Agent 重跑，不要自己按长文档逐步试。

## Agent 应为您执行的命令（参考）

```bash
cd app-factory
npm run build
npm run start -- -p 3001          # 终端 A，Agent 托管
npm run inngest:dev:3001          # 终端 B，Agent 托管
# 必要时 curl 触发重新生成
npm run accept                    # 或 npm run accept -- <项目ID>
```
