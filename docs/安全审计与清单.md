# 安全审计与清单（App 生产工厂）

> **定位**：把「安全原则、当前代码审计结论、分阶段落地清单」写清，供评审与 Cursor 只读审计对照。  
> **纲要级摘要**：[开发纲要.md](./开发纲要.md) 第四节。

---

## 一、核心结论（与「网站是否不安全」无关）

- **安全取决于**：HTTPS、登录与鉴权、密钥只放服务端、数据库行级权限（RLS）、后台任务与生成接口防滥用、日志不泄露密钥、备份与审计。  
- **网址**只是门牌号；**账号、Token、权限、Service Role** 才是门锁与保险柜。  
- **原生 App 不自动更安全**：抓包、反编译、Token 泄露同样存在；密钥仍应放后端。

---

## 二、算力与数据流（回答「大脑是谁」）

| 层级 | 谁提供算力 / 存储 |
|------|-------------------|
| **页面与 API 编排** | 部署环境（本地 Mac / 未来 Vercel 等）上的 Next.js |
| **大模型推理** | 外部 API（如 OpenAI 兼容端点）；本仓库通过服务端 `lib/llm.ts` 调用，**非浏览器直连** |
| **持久化** | Supabase（PostgreSQL） |

## 三、当前仓库审计结果（截至文档编写时的静态结论）

### 3.1 已符合的底线（开发阶段）

| 项 | 结论 |
|----|------|
| `.gitignore` 忽略 `.env*.local`、`.env` | ✅ 降低密钥误提交风险 |
| `OPENAI_API_KEY` 无 `NEXT_PUBLIC_` 前缀 | ✅ |
| `SUPABASE_SERVICE_ROLE_KEY` 仅 `lib/supabase.ts` → 服务端 API / workflow / Inngest | ✅（未发现客户端组件引用） |
| `lib/llm.ts` 仅从服务端工作流侧使用 | ✅（未发现 `import` 进 `"use client"` 组件） |
| 代码库内未见 `console.log` 打印密钥模式 | ✅（仍需避免今后在错误处理中打印整段 `error` 对象含敏感头） |

### 3.2 已知风险（必须在「公网正式上线」前消除或缓解）

| 风险 | 说明 | 严重度（公网场景） |
|------|------|-------------------|
| **无用户登录** | `GET/POST /api/projects`、`GET /api/projects/[id]`、`POST .../generate` 等**未校验「当前用户是谁」** | **高**：任何人可枚举/调用则可读列表、建项、耗 AI 额度 |
| **`projects` 无 `owner_id` / `user_id`** | 表结构无法与 Supabase Auth 用户绑定 | **高**（与上条联动） |
| **未启用 RLS** | 当前使用 Service Role 在服务端通读通写；一旦密钥或 SSRF 出问题，影响面大 | **中高**（上线后应转向「浏览器仅用 anon + RLS」，Service Role 仅服务端受限使用） |
| **Inngest 生产环境** | 须配置 Signing Key，防止伪造同步请求；并对「谁触发了生成」做业务层鉴权 | **中高** |
| **无限流 / 无限额** | 易被刷接口导致模型费用暴涨 | **中高** |
| **局域网调试 `http://IP:3000`** | 仅同 Wi‑Fi 可达；勿向不可信网络暴露、勿提交 `.env.local` | **低**（本地习惯风险） |

> **MVP 自评**：当前适合 **本机 / 可信内网 / 短期演示**；**不适合**在无其它网关保护的情况下长期对公网开放。

## 四、分阶段清单（前期规划 → 中期执行 → 后期落地）

### 4.1 前期（规划已写入纲要；开发中必须守住）

- [ ] 任何密钥**不得**加 `NEXT_PUBLIC_`；不得把 Service Role、OpenAI Key 写进前端或公开仓库。  
- [ ] `.env.local` 不截图外传；不提交 Git。  
- [ ] AI 调用**只经**自有后端 API / 后台任务，**禁止**浏览器持 Key 直连模型商。  
- [ ] 本地调试：不连不可信公共 Wi‑Fi；调试完可停服；防火墙按需开启。

### 4.2 中期（v1.2 联调通过后建议安排）

- [ ] **只读安全审计**（对照本文第三节 + 下文「上线前」子集），输出差异与优先级。  
- [ ] 设计 **Supabase Auth** 登录方式（邮箱 / Magic Link / OAuth 等）。  
- [ ] 数据模型：`projects`（及关联表）增加 **`user_id`**（或 `owner_id`），迁移脚本与回填策略。  
- [ ] 编写 **RLS 策略** SQL（或 Supabase Dashboard 策略说明落盘到 `sql/`）。  
- [ ] API：创建 / 查询 / 生成 / 导出 均校验「资源属于当前会话用户」。  
- [ ] Inngest：生产环境 Signing、事件 payload 最小化；生成入口与业务鉴权一致。  
- [ ] **限流**：按用户 / IP / 项目维度的生成次数与冷却时间。  
- [ ] 日志规范：禁止记录 API Key、完整 Authorization 头、用户密码。

### 4.3 后期（上线前**必须**落地）

- [ ] 正式域名 **HTTPS**（如 Vercel 默认证书）。  
- [ ] **RLS 已启用**且策略经测试：用户 A 无法读写用户 B 的 `projects` / `agent_runs`。  
- [ ] 前端（若存在）仅用 **`NEXT_PUBLIC_SUPABASE_ANON_KEY`** + 用户 JWT；**禁止**在浏览器使用 Service Role。  
- [ ] 生成类接口：**登录 + 额度 + 频率** 双重限制。  
- [ ] Inngest：**Signing Key** 校验、异常告警。  
- [ ] **数据库备份**与恢复演练（Supabase 项目级备份策略）。  
- [ ] 若有 `/admin`：**独立角色校验** + 操作审计日志；敏感操作 MFA。  
- [ ] **隐私政策**与用户数据处理说明（含第三方 AI、日志保留周期）。
